## 3. Core Functional Requirements (Backend Core)

### Module A: The Strict Position Sizing & Capital Guard

Before any trade generated by the Signal Agent can be queued, it must pass through a strict mathematical filter.

* **Inputs:** Ticker, Current Total Balance (€), Planned Entry Price (€), Planned Stop-Loss (€), Planned Take-Profit (€), Target Account Risk % (Default: 1.0%).
* **Engine Calculations:**
* $\text{Allowed Account Risk (EUR)} = \text{Current Balance} \times (\text{Risk \%} / 100)$
* $\text{Risk Per Share (EUR)} = \text{Entry Price} - \text{Stop-Loss Price}$
* $\text{Raw Share Quantity} = \lfloor \text{Allowed Account Risk} / \text{Risk Per Share} \rfloor$
* $\text{Capital Required (EUR)} = \text{Raw Share Quantity} \times \text{Entry Price}$


* **The "Capital Ceiling" Override Rule:** If $\text{Capital Required} > \text{Available Cash Balance}$, the engine must overwrite the raw quantity and cap the position size to $\lfloor \text{Available Cash} / \text{Entry Price} \rfloor$, logging a warning to prevent margin usage.
* **Asymmetry Constraint:** $\text{Risk-to-Reward Ratio (R:R)} = (\text{Take-Profit} - \text{Entry}) / (\text{Entry} - \text{Stop-Loss})$. If $R:R < 1.5$, flag a soft-block warning.

### Module B: The Frictional Drag & German Tax Engine (*Kapitalertragsteuer*)

* **Friction Logging:** Every entry and exit automatically appends a default €1.00 broker execution fee (customizable per broker profile) plus a flat 0.02% capital slippage buffer.
* **Tax Pipeline Logic:**
* Maintain a running calendar-year tally of `Realized Net Gains` minus `Realized Net Losses`.
* First €1,000 of net calendar profit sets `Tax Liability = €0` (The German *Sparerpauschbetrag* allowance).
* Every euro profit after the €1,000 threshold triggers an automatic **26.375%** tax withholding calculation, showing a true "Net Liquid Balance" alongside the gross equity curve.

## 4. Advanced Edge-Case State Machine Requirements

To prevent standard market anomalies from breaking data metrics, trades must transition through strict state paths:

### State Path 1: The "Sideways Base" Protocol (Stagnant Stalls)

* **Rule:** If a stock triggers a stop-loss, that trade database object is permanently marked `CLOSED`. If that stock consolidates flat or moves sideways right near or slightly below that support level for 2+ days, it **cannot** be modified or "re-opened".
* **Re-Entry Implementation:** To trade it again, the asset must be qualified as a completely separate database entry with a fresh unique ID. It requires an automated breakout trigger candle (closing above the high of the 2-day consolidation zone) before a new position window is generated.

### State Path 2: Time-Outs & Trailing Stops (Target Unfulfilled)

The application must monitor open positions via background cron jobs to identify stuck capital. The database `trade_logs` table must support four strict `exit_reason` codes:

1. `TARGET`: Profit target hit perfectly.
2. `STOP`: Stop-loss floor breached.
3. `TIME_OUT`: Stagnancy timeout tracker. If a trade remains open for **10 consecutive trading days** without hitting either bracket boundary, trigger a manual exit alert to reclaim dead capital.
4. `TRAILED`: Once price travels 50% of the distance to the profit target, the database object must automatically update its active stop-loss field to the **Entry Price (Break-Even)** to eliminate downside risk.

## 5. Automated Notification & Safeguard Engine

The system needs an asynchronous cron layer to push alerts regarding critical lifecycle markers:

* **The Earnings Barrier:** Ingest upcoming market corporate calendars. Raise a high-priority push notification or UI alert 3 days prior to an earnings release for any ticker on the active watchlist or portfolio.
* **The Dividend Offset:** Alert the user 48 hours before an Ex-Dividend date to adjust technical stop-losses, preventing artificial price gaps from triggering liquidations.
* **Technical Scanners:** Daily polling using `yfinance` data to flag assets whose Daily RSI drops below 30 (Oversold Swing Candidate).

## 6. Database Schema Reference (SQLAlchemy)

```python
from sqlalchemy import Column, Integer, String, Float, DateTime
from sqlalchemy.ext.declarative import declarative_base
import datetime

Base = declarative_base()

class TradeLog(Base):
    __tablename__ = 'trade_logs'
    
    id = Column(Integer, primary_key=True, autoincrement=True)
    ticker = Column(String(10), nullable=False)
    status = Column(String(10), default='OPEN') # OPEN, CLOSED
    exit_reason = Column(String(15), nullable=True) # TARGET, STOP, TIME_OUT, TRAILED
    
    # Timeline
    entry_date = Column(DateTime, default=datetime.datetime.utcnow)
    exit_date = Column(DateTime, nullable=True)
    days_in_trade = Column(Integer, default=0)
    
    # Financial Boundaries
    share_quantity = Column(Integer, nullable=False)
    entry_price = Column(Float, nullable=False)
    original_stop_loss = Column(Float, nullable=False)
    active_stop_loss = Column(Float, nullable=False) # Dynamically moves for Trailing/Break-Even
    take_profit = Column(Float, nullable=False)
    actual_exit_price = Column(Float, nullable=True)
    
    # Overhead Costs
    broker_fees = Column(Float, default=2.00) # Combines entry + exit
    slippage = Column(Float, default=0.00)
    
    # Realized outputs computed instantly on trade closure
    gross_pnl = Column(Float, default=0.00)
    net_pnl_after_fees = Column(Float, default=0.00)
    tax_withheld = Column(Float, default=0.00)
    final_liquid_pnl = Column(Float, default=0.00)

```
